# 审查报告：biomed-research-integrity-auditor

> 审查日期：2026-06-29
> 审查方式：通读全部源码（约 6,300 行 Python）+ 实际运行流水线 + 注入式攻击复现 + 测试套件执行
> 审查立场：客观找不足，结合真实使用场景。优点只作背景，主体是问题与风险。

---

## 0. 一句话结论

这是一个**工程纪律优秀、定位克制、文档完善**的"研究诚信风险审计"骨架——分层（detector → 契约 → 校准 → 报告）、不下不端结论、风险上限机制都做得专业。但它目前**更像一套"评测合成数据能跑通"的可信演示，而不是能审真实生物医学论文的工具**。两个根本问题：

1. **检测器在真实数据上要么打不准、要么打不动**（图像检测算法覆盖窄且复杂度爆炸，统计筛查在真实小样本表上会"逢表必响"）。
2. **整套"可信"机制建立在"作者自报即真相"之上**——一行作者自己写的 assembly manifest 就能把真实的图像重复发现从 R3 压到 R1（已实测复现）。

下面按"真实使用场景"展开。

---

## 1. 场景一：实验室拿自己 50 张高清图的论文做投稿前自查

这是 README/SKILL 主打的 `Presubmission Internal Audit` 场景。结论：**这个场景目前基本跑不动，且会漏掉最常见的造假类型。**

### 1.1 图像检测算法的覆盖面太窄（会漏真问题）

`detectors/image/` 对标的是 ImageTwin / Proofig / Forensically 这类工具，但实际只覆盖了"整图近重复"这一窄类：

- **完全没有"同图内 copy-move"检测**。`local_patch_reuse.py:415` 的循环是 `images[i+1:]`，永远只比较**不同**图，从不拿一张图和自己比对。而"复制一块细胞/背景盖掉瑕疵"恰恰是最常见的图像操纵类型之一——本工具结构上无法发现。
- **裁剪、局部拼接、任意角度旋转全部漏检**。三种 hash（aHash/dHash/pHash）都把整图缩到 8×8 或 32×32（`global_near_duplicate.py:30-101`），裁剪/缩放后的复用整图 hash 直接对不上；`TRANSFORMS`（`:14-23`）只覆盖 90° 旋转和翻转，5°/17° 旋转这种造假者随手就能做的变换完全不在范围内。
- **66%–99% 的局部复用掉进两个检测器中间的缝**。`local_patch_reuse.py:299` 把匹配区域 > 65%（`max_region_fraction=0.65`）的候选直接丢弃，理由是"整图检测器会接住"——但整图检测器只在无裁剪无位移时才接得住（见上一条）。一个 70% 重叠的裁剪复用，两个检测器都不报。
- **缺乏 keypoint/特征匹配**，所以对任意旋转、缩放、透视没有鲁棒性；没有拼接取证（ELA、JPEG ghost、噪声/CFA 不一致）。这些是商业工具的核心能力，本项目从原理上不具备。

### 1.2 真实显微图像格式会被悄悄毁掉或丢弃

- `convert("RGB")`（`global_near_duplicate.py:186`、`local_patch_reuse.py:403`）对 16-bit TIFF 显微图（PIL 模式 `I;16`/`I`/`F`）不是抛错就是把动态范围截断到低 8 位——喂给 hash/NCC 的亮度本身就是错的。而 16-bit TIFF 正是这个工具声称要审的"原始显微数据"。
- 多帧/Z-stack/多通道 TIFF 只读第 0 帧，其余切片从不参与比对。
- 出错被 `except Exception` 吞掉只计数（`global:195`、`local:410`）。50 张图里 30 张静默加载失败，CLI 只打印 `errors: N`，审计者会**误以为全部图都筛过了**。对一个诚信工具，"静默的部分覆盖"是实打实的危险。

### 1.3 复杂度爆炸，50 张高清图根本跑不完（实际可用性）

`local_patch_reuse.py` 是纯 Python 的 O(N²·T²·8)，内层还有 16,384 像素的纯 Python NCC 循环：

- `best_tile_match`（`:148-173`）对每个左 tile 都把右 tile 的 8 个变换+hash 重算一遍——本可缓存，实际是 225× 冗余。
- 一对 2048² 显微图 ≈ 920K tile 对 × 8 变换 ≈ 740 万次操作；50 张图 = 1,225 对，总量是**数亿到数十亿次纯 Python 操作**，量级是小时到天，不是交互式。
- 内存：每张图保留 RGB 副本 + 上千个 128×128 PIL crop，50 张高清图 ≈ 700MB+，真实 OOM 风险。

**也就是说，这个工具恰恰在它主打的"高分辨率、多面板真实论文"输入上不可用。**

## 2. 场景二：统计学家喂入真实 qPCR / Western / 动物肿瘤 CSV

`stats_consistency_check.py`（835 行，最大的检测器）。结论：**在真实小样本表上会"逢表必响"，把真信号埋进噪声里。**

### 2.1 阈值是为浮点噪声、不是为"报告精度"设计的（高频误报）

- **SD = SEM·√n 容差 `1e-3`（`:182`）太紧**。作者把 SEM/SD 只报到 2–3 位有效数字，光是输入的正常四舍五入就能让 0.1% 相对容差爆掉。这个筛查会在**大量合法、正确舍入**的汇总表上误报。
- **整数计数的 GRIMMER 式平方和检查（`:145-160`）假设 SD 是精确值**，但 SD 是舍入后的；舍入带来的不确定度比 `0.02` 的容差大约 1000 倍——几乎对任何舍入过的计数汇总表都误报。
- **终末数字筛查在小 n 上按设计就会误报**：`effective_min_count` 对 ≤5 个值返回 3，`dominance` 默认 0.65；3 个值里 2 个末位相同 = 0.667 ≥ 0.65 直接报。qPCR/Western 每组 3–6 行是常态，3 个值里 2 个末位相同纯属偶然。终末数字/Benford 分析在 < 20–30 个值时统计上无意义，把门槛设成 3 站不住脚。
- **舍入、精度混合、同秩、相关性 ≥ 0.995（`:299-548`）**这一组筛查在真实数据上几乎必触发：组织学评分（0/0.5/1/1.5/2）、标准曲线、技术重复、肿瘤体积 vs 肿瘤重量这种生物学天然相关的列，全都会被点名。

### 2.2 领域误判：CD4/CD8 被当成时间点

`TIME_TOKEN_RE`（`:19`）含无词边界的 `d\d+`/`h\d+`，于是 `time_token("cd4")="d4"`、`cd8→d8`、`cd3→d3`。`check_time_stratified_shifts`（`:605`）据此**选列**，结果流式细胞和免疫学最常见的 CD4/CD8/CD3/CD45、组蛋白 H2/H3 列被当作纵向时间点喂进趋势筛查。纯粹的领域级误触发。

### 2.3 SKILL 宣称、代码没有的筛查（文档与实现不符）

- SKILL（`:88,131,164`）反复列 **"Benford-style"** 筛查——**全文件没有任何 Benford 实现**，只有单一末位数字占比。
- SKILL 反复列 **"p-value clustering"**——代码只做了 `0≤p≤1` 的范围检查（`:199`），没有任何分布/聚集分析。
- SKILL（`:187`）写明支持 **CSV/XLSX**，但 `CSV_EXTS={".csv",".tsv"}`（`:16`）静默丢弃 `.xlsx`。审计者指向一个 Excel 导出表会得到**零发现且零报错**。

### 2.4 伪重复检测器形同虚设

`pseudoreplication_screen.py:71` 的核心触发条件是"技术单元数 > 生物单元数"——这正是**规整长表的正常形态**（每只鼠 3 个孔）。它报的是**数据布局**，不是分析错误；唯一能区分真问题的 `reported_n_basis` 列真实数据里基本不存在，所以那个判据实际是死的。它把"数据里有技术重复"误等同于"分析犯了伪重复"。

### 2.5 本地化数据直接被算错

`parse_float`（`:42`）用 `replace(",", "")`：欧洲格式 `"1,5"` 变成 `15.0`（差 10 倍），`"1,234"` 变成 1234。非美区导出的 CSV 在生物医学很常见，所有下游计算无声地建立在错了一个数量级的数上。

## 3. 场景三：有人想"骗过"这个审计器（最严重）

这是诚信工具最该扛住的对抗场景。**实测：扛不住。**

### 3.1 一行作者自报的 manifest 就能把真发现压掉（已复现）

我把 `case_004`（重复/翻转面板，基准答案 R3）原样复制，只往 `figure_assembly/assembly_manifest.csv` 里加了一行，声称两张被判重复的图"互为来源"：

```csv
figure_panel,source_record,relation_type,modality,notes
figures/Figure_2B.png,figures/Figure_4D.png,declared_derived_from,microscopy,same field reused
```

重跑流水线，**overall_risk 从 R3 掉到 R1**，原本的图像重复发现被改写成 `unresolved_fig_raw_similarity` 完整性缺口。

根因在 `contextual_joiner.py:167-175`：只要候选的边落进 `declared_pairs`，就被标成 `expected_traceability` / `positive_evidence` 并整条丢弃。而 `declared_pairs` 完全来自**作者自己写的 manifest**，系统：

- 不做任何核验（不比 hash、不比像素、不看元数据）——作者说"这俩同源"就当真；
- **无视它自己算出来的 `confidence`**（`build_resource_graph.py:96` 算了，`contextual_joiner` 压制路径根本不读），`confidence:0.0` 和 `0.98` 压制效果一样；
- **无视 `relation_type`**——`parse_assembly_manifest.py:89` 把所有行硬编码成 `expected_traceability`，哪怕写 `relation_type: totally_different_experiment` 也照样压制；
- 给作者自报的字符串**硬编码 0.98 置信度**（`:93`）。

更夸张的是 `infer_ordered_links`（`:180-207`）：任何文本 manifest 只要包含魔法短语 `"figure panels map to"` 外加数量匹配的 raw 文件名列表，就**按文件名排序**自动给每张图建压制链接。塞进短语 + 对数量即可批量制造"洗白"。

### 3.2 风险上限全部建立在候选"自报标签"上

`risk_cap_engine.py:119` 的 R4 闸门检查的是候选自己的 `risk_cap_tags`/`evidence_strength`。一个候选只要自报 `evidence_strength:"direct_contradiction"`，既能拿到 R4 建议、又同时满足 R4 闸门——引擎不做任何结构性交叉核验。反向也成立：本该封顶 R3 的图像候选，只要额外带一个 r4_requirements 标签，`unless_r4_requirement` 上限就被跳过，直冲 R4。

### 3.3 提示注入边界写在文档，但压制逻辑给了更隐蔽的注入面

SKILL 正确地提醒"manifest 里的文字是材料不是指令"。但真正的攻击面不是自然语言注入，而是**结构化字段注入**——上面 3.1 的 CSV 行就是合法字段，绕过了所有"忽略 manifest 里的指令"防御。

## 4. 横切问题：可信机制的"静默降级"路径

`scripts/audit_package.py` / `calibrators/` 里有多处"看起来安全、实际可被无声绕过"：

- **空检测器集 = 干净放行**（`audit_package.py:237`）。检测器按硬编码扩展名 gating（`SOURCE_EXTS={".csv",".tsv"}` 等）。包里数据若用别的扩展名，所有检测器跳过，输出 `findings:[]`——与"审过且干净"**无法区分**。这是最危险的静默失败：不可分析的包长得和通过的包一模一样。
- **jsonschema 缺失时契约校验静默降级**（`contract_validation.py:39`）。`except Exception`（不是 `ImportError`）后走 `_fallback_validate_required`，只查顶层字段是否存在——一个 `calibrated_risk_level:"R9"` 的发现照样通过，且**不打任何警告**。
- **legacy `findings` 入口完全绕过检测器 schema 校验**（`risk_cap_engine.py:213`）。手写一个 `findings` 负载即可跳过检测器契约和整个 provenance 校准，`risk_level` 被直接抄进 `risk_suggestion`。
- **R3+ 强制字段被自动填充**（`risk_cap_engine.py:151-160`）。缺 benign_explanations/required_materials 时引擎**自己编**样板文字，于是"R3+ 必须考虑良性解释"这条规则从不拦截，只是糊上去。一个 R4 发现可以带着从没真正考虑过的良性解释发货。

### 4.1 risk_rules.yaml 里有"安全规则"引擎根本不读

- `external_public_material.missing_source_data_max: R1`（`risk_rules.yaml:6`）——引擎只读 `default_max`（`risk_cap_engine.py:122`），这条"外部材料缺源数据则封顶 R1"的规则**什么都没做**，外部发现实际封在 R3。
- `expected_traceability.report_as: positive_evidence`（`:53`）——`cap_from_rule` 只认 `max`/`default_max`，对 `report_as` 返回 `None`，引擎层面这条指令被忽略（只有上游 joiner 兑现）。
- `local_patch_within_declared_raw_source`、`local_patch_direct_source_conflict`、`external_public_material_only` 等键，contextual_joiner 从不产出对应标签，是**够不到的死键**。

文档（architecture.md）把这些都当作已生效的安全特性描述，与实现不符。

## 5. 场景四：把这套 eval 当作"质量已被验证"的依据

**这是对外宣传里最需要打折扣的一点。**

- **从未有过 LLM 评测结果**。`evals/scorecards/` 只有 `.gitkeep`，`evals/outputs/` 只有一个 `script_baseline` 子目录。README 描述的"对接 agent 跑 30 个 case 再 score"的闭环**从未实际跑过并留痕**。
- **合成数据不代表真实分布**。所有 `manuscript.pdf` 其实是 **ASCII 纯文本**（`file` 确认 30/30 都是 `Title:...` 开头的文本），不是真 PDF；图像是 `ImageDraw` 画的 180×140 椭圆/方块（`generate_synthetic_cases.py:38-92`），没有真实显微图的噪声、位深、压缩特征。检测器在这些"玩具图"上的表现，对真实 16-bit TIFF 没有外推价值。
- **CI 不校验审计结果**（`.github/workflows/validate.yml`）。"key audit regressions" 只跑 `audit_package.py ... `并依赖 `check=True` 退出码——**只验证"没崩"，从不断言风险等级正确**。`make validate` 同理（py_compile + unittest + 生成 prompt）。一个把所有 case 都判 R0 的回归不会被 CI 发现。
- 真 PDF benchmark 诚实地把 PDF 文本抽取标为"已知缺口"——这点是加分的克制，但也意味着**真实论文最常见的输入（二进制 PDF）目前完全无法处理正文/图注/图像抽取**。

### 5.1 单检测器崩溃会拖垮整次审计

`audit_package.py:32` 的 `run()` 用 `subprocess.run(check=True)` 且全程无 try/except——任一检测器非零退出或吐坏 JSON，整次 run 直接塌掉，没有部分结果隔离，唯一诊断是子进程 stderr（还没被捕获）。

## 6. 其它工程问题（较次要）

- **版本号漂移**：`pyproject.toml` 是 `0.2.0`，CHANGELOG 已经到 `v0.4.0` + Unreleased，README 引用 0.3.2/0.4.0 特性。对外发布会让人困惑实际能力。
- **死代码/无效守卫**：`local_patch_reuse.py:269` 的 `seen_regions` 去重是 no-op；`stats_consistency_check.py:116` 的 `almost_equal` 从不被调用；多处 `candidate_id` 先赋值后被重编号覆盖。
- **扩展名集跨模块不一致**：`audit_package.py:27` 与 `contextual_joiner.py:22` 的 `SOURCE_EXTS` 不同，导致 `.xlsx` 不被统计检测器分析，却能让 `has_source_data()` 为真、从而**白拿一个 disclosed_legitimate_reuse 宽松上限**。
- **`resize()` 不指定重采样滤波**（`global:31` 等），与 imagehash 参考实现（LANCZOS）不一致，hash 不可复现、抗缩放更弱。
- **`<` 剥离**（`parse_float:45`）把检测限值 `"<5"` 当成 `5`，无声扭曲统计。

## 7. 优先级建议（按"提升真实可信度"排序）

**P0 — 不修就不能称"可信审计"：**
1. provenance 压制必须可核验：声明的 figure↔raw 关系要做像素/hash 交叉核验，至少要读 `confidence` 和 `relation_type`，并对"作者自报且无法核验"的关系**降级为 R1 待核**而不是直接压成 positive evidence（`contextual_joiner.py:133-175`）。
2. 区分"跳过/不可分析"与"审过且干净"：空检测器集不能输出等价于 R0 的结果（`audit_package.py:237`）。
3. 契约校验在 jsonschema 缺失时**失败/告警**，而不是静默降级（`contract_validation.py:39`）；R3+ 强制字段缺失时降级或拦截，而不是自动编（`risk_cap_engine.py:151`）。
4. 把 `risk_rules.yaml` 里"写了但没读"的安全键真正接进引擎，或删掉以免误导（`missing_source_data_max` 等）。

**P1 — 让检测器在真实数据上有意义：**
5. 统计阈值改为"基于报告精度"而非浮点容差；终末数字/Benford 加最小样本量门槛；修 CD4/CD8 时间列误判；补齐或删掉文档宣称的 Benford / p-value clustering / XLSX 支持。
6. 图像检测：加同图内 copy-move；补 keypoint 特征匹配以覆盖任意旋转/缩放/裁剪；正确处理 16-bit/多帧 TIFF；加载失败要显著上报而非吞掉。
7. 局部 patch 检测的复杂度与内存优化（缓存右 tile 变换、限制分辨率/分块），否则主打场景跑不动。

**P2 — 让"已验证"名副其实：**
8. 真跑一轮 LLM eval 并把 scorecard 落库；CI 增加对 overall_risk / 关键发现的断言，而非只看退出码。
9. 用少量**真实**（或高保真合成）显微图 + 真 PDF 扩充评测集；统一版本号。

## 8. 总评

| 维度 | 评价 |
| --- | --- |
| 定位与边界（不下不端结论、R0–R4、克制语言） | 优秀，是这个项目最大的价值 |
| 架构分层（detector/契约/校准/报告 分权） | 设计专业，思路正确 |
| 文档完整度 | 高（但部分描述领先于实现） |
| 检测器对真实数据的有效性 | **弱**：图像覆盖窄+跑不动，统计高频误报 |
| 对抗鲁棒性（可信的核心） | **不足**：作者自报即真相，实测可一行洗白 |
| "已验证"的可信度 | **被高估**：无 LLM eval 实绩，合成数据非真实，CI 不验结果 |

一句话：**好骨架，但"可信"二字目前主要靠文档和合成 demo 支撑，而非真实数据与对抗测试的证据。** 把 P0 的"作者自报即真相"和"静默降级"补上，再用真实数据验证检测器，它才能从"可信演示"走向"可信工具"。
